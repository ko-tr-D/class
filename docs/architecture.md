# Architecture

## 시스템 구성

```text
Browser
  -> Teacher Web App
  -> Student Web App
    -> API Server
      -> PostgreSQL
      -> Object Storage
      -> Background Worker
        -> OCR Provider
        -> AI Provider
```

Teacher Web과 Student Web은 같은 웹앱 안에서 역할별 화면으로 분리합니다. API Server는 모든 데이터 접근을 통제하고, 브라우저가 DB에 직접 접근하지 않게 합니다.

## 도메인 모듈

- `auth`: 로그인, 역할, 반/학년도 권한, 학생 접속 코드
- `ingestion`: PDF 업로드, OCR, 텍스트 검수, 수업 자료 태깅
- `curriculum`: 학년도, 반, 단원, 성취기준, 평가 기준
- `students`: 학생, 반 배정, 학생 특성 기록
- `evaluation`: 루브릭 평가, 근거 연결, 피드백
- `records`: 생활기록부 문구 초안, 교사 수정 이력
- `assessment`: 문항, 시험지, 학생 풀이, 채점
- `analytics`: 성취도, 오답 유형, 풀이 패턴, 알고리즘 버전

## 데이터 흐름

1. 교사가 PDF를 업로드합니다.
2. 원본 파일을 저장하고 OCR 작업을 큐에 넣습니다.
3. OCR 결과를 페이지/문단 단위로 저장합니다.
4. 교사가 텍스트를 검수하고 단원, 날짜, 성취기준을 연결합니다.
5. 평가 기준에 따라 학생별 근거와 성취 수준을 기록합니다.
6. 생활기록부 문구 초안을 생성하고 교사가 수정/승인합니다.
7. 학생용 평가는 풀이 로그를 저장하고 분석 모듈이 패턴을 계산합니다.

## AI 사용 위치

- OCR 후 텍스트 정리 제안
- 필기 내용의 키워드/활동 단위 분류
- 평가 근거 요약
- 생활기록부 문구 초안 작성
- 문제 풀이 패턴 설명 초안

AI는 자동 판단자가 아니라 교사의 판단을 보조하는 도구로 둡니다.

## 개인정보/보안

- 학생 식별 정보와 학습 기록을 분리 가능한 구조로 설계합니다.
- 학년도 단위 보관/이관/삭제 정책을 문서화합니다.
- 모든 생활기록부 초안은 생성 근거와 수정 이력을 저장합니다.
- 학생용 접속은 평가별 제한 코드 또는 학교 계정 연동으로 시작합니다.
- 모든 민감 데이터 접근은 감사 로그에 남깁니다.
- AI 요청 전 학생 이름, 번호 같은 직접 식별자는 제거합니다.
