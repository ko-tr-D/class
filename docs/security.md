# Security Model

이 프로젝트는 학생 개인정보, 평가 기록, 교사 관찰 기록, 생활기록부 문구 초안을 다룹니다. 따라서 기능 개발보다 먼저 보안 원칙을 고정합니다.

## 기본 원칙

- 브라우저는 API 서버를 통해서만 데이터에 접근합니다.
- 데이터베이스는 외부에 직접 노출하지 않습니다.
- 교사, 학생, 관리자 권한을 분리합니다.
- 학생 개인정보와 학습 기록은 가능한 한 분리 저장합니다.
- 민감 데이터 접근과 수정은 로그로 남깁니다.
- AI 처리에는 필요한 최소 정보만 보내고 직접 식별자는 제거합니다.

## 권한 모델

### Teacher

- 이메일과 비밀번호로 로그인
- 자신에게 배정된 학년도, 반, 학생 자료 조회
- 수업 자료 업로드와 OCR 검수
- 평가 기준 생성과 평가 기록 작성
- 생활기록부 문구 초안 생성, 수정, 승인

### Student

- 교사가 배포한 접속 코드로 배정된 평가에만 접근
- 본인의 풀이 제출
- 허용된 경우 본인 피드백 확인

### Admin

- 교사 계정 관리
- 학교/학년도/반 설정
- 보안 로그와 백업 정책 관리

## 데이터 분류

- Public: 앱 안내 문구, 공통 코드
- Internal: 단원, 성취기준, 평가 기준표
- Sensitive: 학생 이름, 번호, 반 배정, 평가 결과, 관찰 기록
- Highly Sensitive: 생활기록부 문구 초안/최종본, 교사의 서술형 관찰 기록

## 필수 보안 기능

- HTTPS 강제
- 비밀번호 해시 저장
- 세션/토큰 만료
- 역할 기반 접근 제어
- 반/학년도 단위 데이터 접근 제한
- DB 백업 암호화
- 민감 테이블 접근 로그
- 파일 업로드 확장자/용량 제한
- PDF 원본 파일 접근 권한 검사
- AI 요청 익명화

## 로그인 정책

- 교사: 이메일 로그인
- 학생: 평가별 접속 코드
- 학생 접속 코드는 원문을 저장하지 않고 해시로 저장합니다.
- 학생 접속 코드는 만료 시간, 사용 여부, 배정 평가를 함께 검사합니다.

현재 API 구현에서는 교사 비밀번호와 학생 접속 코드를 해시로 저장하고, AI 문구 생성 요청에는 학생 이름과 번호를 제거한 익명화 payload를 `ai_requests`에 남깁니다.

## 감사 로그 예시

- 교사가 학생 기록을 조회함
- 교사가 생활기록부 문구 초안을 생성함
- 교사가 최종 문구를 수정/승인함
- 학생이 평가를 시작/제출함
- 관리자가 계정 권한을 변경함

## AI 사용 규칙

- AI는 평가자나 최종 작성자가 아니라 초안 작성 보조 도구입니다.
- AI 요청에는 학생 이름, 번호를 직접 포함하지 않습니다.
- AI 응답은 교사 승인 전까지 공식 기록으로 보지 않습니다.
- AI가 사용한 근거 자료 ID를 함께 저장합니다.
